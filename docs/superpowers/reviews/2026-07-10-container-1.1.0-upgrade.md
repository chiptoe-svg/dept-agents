# Apple Container 0.12.3 ‚Üí 1.1.0 upgrade ‚Äî evidence

**Date:** 2026-07-10. Plan 2, Task 4. Operator authorized stopping both installs.

## Result: PASS

Department install verified live on 1.1.0: a real agent turn replied `upgraded`, container on `192.168.65.3/24`, zero proxy 401s, no new errors.

## What the upgrade actually changed (beyond the release notes)

1. **`status` became an object.** `container ls --format json` now returns
   `status: { state: "running", networks: [...], startedDate: ... }` instead of the bare
   string `"running"`. Probed live ‚Äî the shape matches exactly what Task 3's
   `containerState()` was written to accept, so orphan reaping survives.

2. **The bridge subnet moved: `192.168.64.0/24` ‚Üí `192.168.65.0/24`.** This is NOT in the
   release notes and it broke a hardcoded constant. `detectHostGateway()` read
   `bridge100`/`bridge0` and fell back to the literal `192.168.64.1`. On 1.1.0 there is no
   `bridge100` **unless a container is running**, and `bridge0` has no IPv4 ‚Äî so a host
   starting with no containers resolved the dead `192.168.64.1` and handed it to every
   container as `ANTHROPIC_BASE_URL` / `OPENAI_BASE_URL` / `GWS_MCP_RELAY_URL`. Every LLM
   call would have failed.

   Fixed in `488da89f`: `CONTAINER_HOST_GATEWAY` is now a memoized function with the
   precedence ‚Äî (1) `CONTAINER_HOST_GATEWAY` env/`.env` override, (2) query the runtime
   (`container network inspect default` ‚Üí `[0].status.ipv4Gateway`), (3) interface scan
   (`bridge100`/`bridge0`, back-compat with 0.12.x), (4) **throw**, naming all three
   attempts. Deliberately no hardcoded fallback: a wrong-but-plausible constant is what
   turned a network change into a silent total outage.

   Live check: `CONTAINER_HOST_GATEWAY()` ‚Üí `192.168.65.1`, matching
   `container network inspect default`.

## Rollback status ‚Äî degraded, read this before assuming you can undo it

`brew upgrade` **deleted** the 0.12.3 keg *and* its cached bottle
(`Removing: /opt/homebrew/Cellar/container/0.12.3... (20 files, 263.2MB)`), so the rollback
prepared before the upgrade no longer exists. Reverting now means refetching the old
formula from Homebrew history (`brew extract`) ‚Äî possible, but not a one-liner. Only the
1.1.0 keg remains.

## The personal install (`/Users/admin/projects/nanoclaw_personal`) ‚Äî works now, fragile

It shares the `container` binary and daemon, so it was upgraded too. Its own
`src/container-runtime.ts` still has the old `detectHostGateway()` with the hardcoded
`192.168.64.1` and **no env override**.

It is currently healthy ‚Äî both of its containers respawned on `192.168.65.4/24` and
`192.168.65.5/24`, and its containers were handed `192.168.65.1`, because `bridge100`
existed at the moment its host started (a department container was already running).

**Latent failure:** if that host ever starts while *no* Apple Container is running ‚Äî
a reboot, or a restart during a quiet period ‚Äî `bridge100` will be absent, `bridge0` has
no IPv4, and it will fall back to `192.168.64.1`. Its agents will then fail every LLM call
with no obvious cause.

**Fix:** port `488da89f`'s `detectHostGateway()` into that repo (one file, plus its test).
Not done here ‚Äî modifying the operator's other install is their call.
